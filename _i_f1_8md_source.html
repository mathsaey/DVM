3 # IF1 Quick Reference
4 
5 This page serves as a basic introduction to the IF1 language.
6 The information on this page is based on the paper: `IF1 - An Intermediate Form for Applicative Languages. (Stephen Skedzielewski 1985)`
7 
8 On this page, we attempt to create a compact IF1 overview that can serve as a full reference to the language while still being complete.
9 If you like to look at an example while browsing this page, check out [a sorting algorithm in IF1](@ref SortIF1), generated from [this sisal code](@ref SortSis).

13 # IF1 Concepts {#Concepts}
14 
15 IF1 represents directed, acyclic graphs, these graphs have a few components.
16 
17 * IF1 nodes are operations.
18  * a node can have an arbitrary amount of in -and outputs.
19  * IF1 defines over 50 nodes, that is, "native" operations.
20  * The in -and output of a node happens through **ports**, which are numbered indicators of in and output locations
21  * Not every node has a predefined amount of ports.
22 * Edges represent data paths between nodes.
23  * Edges carry **type information**
24  * Not mandatory
25  * Can describe user-defined types and built-in types.
26  * A special edge exists to describe literal constants.
27  * Don't contain a source port
28  * Contain the text of a literal as a string.
29 * Graph boundaries surround groups of nodes and edges.
30 
31 # IF1 Instructions {#Form}
32 
33 IF1 instructions are delimited by newlines. The first character after a newline determines the type of instruction the line represents, the rest of the line consists of a number of whitespace separated fields, the amount of which depends on that specific instruction. Any extra text listed after the fields is considered to be a comment, so are lines that start with C. For more information about comments, look into [appendix A](#Comments).
34 
35 The following table lists the different instructions types.
36 
37 Character | Represents | Syntax
38 ----------| -----------|-------
39 T | type | `T label type_code arg_1 arg_2 `
40 E | edge | `E source_node port destination_node port type `
41 L | literal | `L destination_node port type string `
42 G | subgraph of a compound node | `G type_reference `
43 G | local function graph | `G type_reference "name" `
44 X | global function graph | `X type_reference "name" `
45 I | imported function | `I type_reference "name" `
46 N | simple node | `N label operation `
47 { | start compound node | `{ `
48 } | end compound node | `} label operation association_list_length association_list_element_1 association_list_element_2 ...`
49 
50 
51 The semantics of these instructions are explained below, explanation about *labels* and *types* are presented in the [labels and types](#types) section.
52 
53 * Simple nodes simply contain a label and an identifier, this identifier represents a native IF1 instruction.
54 * Compound nodes contain subgraphs and represent more complex operations, such as a for loop.
55 * The compound node contains a few subgraphs that represent parts of the compound node, such as the initialization and the condition check, the association list shows which subgraphs represent which action.
56 * Graph boundaries represent a function and simply contain the type (check out the function type) and name of this function.
57 * Graph boundaries have a scope associated with them, anything below the boundary declaration is part of the graph's scope until either a "}, G,X or I" is encountered.
58 * Subgraphs in compound nodes are not required to give a type.
59 * Edges simply contain the input node, and the port on this node, the destination node, and the port on this node that this edge leads to. Furthermore, the edge also contains the type of it's data.
60 * Literals contain their destination, the port on this destination, their type and a string version of their counts. [Appendix B](ref #Literal_def) contains an overview of some of the most common literals.
61 
62 ## Labels and Types {#types}
63 
64 IF1 instructions commonly contain a type identifier, the purpose of this section is to explain IF1 type definitions, any defined type is referred to by it's *label*.
65 
66 Labels are represented by integers, and are used to identify nodes and types. Nodes and types do not share labels, that is, a node and a type may have the same integer as label without being related in any way. Type labels share a global scope, while Node labels only have to be unique within their enclosing graph. It's also worth noting that a missing or unknown type is referred to by using the label 0.
67 
68 With that being said, we can look at the type definitions:
69 
70  T label type_code arg_1 arg_2
71 
72 * Label can be used to refer to the type later on.
73 * The type code can be considers to be an argument that indicates what type of type we are dealing with, based on this information, the basic_type is used as another argument to construct the actual type.
74 * The exact use of the other codes depends on the type_code.
75 
76 To clarify this, let's look at a basic sisal types. In this case, the boolean.
77 
78 `T 1 1 0 %na=Boolean`
79 
80 * T implies that we are declaring a type
81 * 1 is the *label* of the type, it will be used to refer to the type we are constructing from now on.
82 * 1 is the *type_code* of the type. This particular type code tells us that we are dealing with a *basic code*.
83 * From the information of the type_code, we know that 0 is a basic code, in this case the basic code 0 represents a boolean.
84 * The rest of the line is a comment, it tells us we are dealing with a boolean.
85

87

89 -----|----------|------------|-----------
90 0 | Array | Base type | ` none `
91 1 | Basic code | Basic code | ` none `
92 2 | Field | Field type | Next field

95 5 | Record | First field | ` none `

103 #### Basic types
104 
105 As the example in the parent section indicates, basic types simply indicate a standard type, the argument is simply a code that tells us which type we are dealing with. A table with the basic types and their code can be found below.
106 
107 Code | Type
108 -----|-----
109 0 | Boolean
110 1 | Character
111 2 | Double
112 3 | Integer
113 4 | Null
114 5 | Real
115 6 | WildBasic*
116 
117 `* This type does not appear in the official reference, but is generated by sisalc 14.1.
118 
119 #### Array and Streams
120 
121 Arrays and streams simply point to the *label* of the type they contain.
122 
123 #### Records and fields, unions and tags. And tuples.
124 
125 A record can be seen as a pointer to a field. It simply contains it's label, the type_code that indicates it's a record and the *label* of it's first field.
126 
127 A field contains it's label, it's type_code, the *label* of the type it contains, and the *label* of the next field. The last field contains 0 as the *label* that points to the next field.
128 
129 Unions and tags work in the same way, the only difference is that unions can be seen as a pointer to a tag. A tag points to the next tag like a field points to the next field.
130 
131 Tuples follow the same contain your own type, point to the next one convention, but don't require an initial pointer.
132 
133 #### Functions
134 
135 A function type simple contains "pointers" to 2 tuples, the first tuple represents the arguments this function accepts while the second tuple represents the result it returns.
136 
137 In terms of higher level languages, the function type simple declares the function's signature, without the name.

140 # Appendix A: Comments {#Comments}
141 
142 2 main types of comments exist in IF1, stamps and pragmas, stamps are comments that occupy an entire line that starts with `C$`, pragmas are comments that are added after the fields of a line.
143 
144 ## Stamps {#Stamps}
145 
146 Stamps are used to mark some processing that has been done to a file, a few stamp type exists. A stamp line has the following syntax: `C$ stamp_type info`.
147 The following table provides an overview of the different stamp types.
148 
149 Character | Meaning
150 ----------|--------
151 A | Array update analysis
152 C | Structure checker
153 D | Order nodes using data dependencies
154 E | Common subexpression information
155 F | Frontend
156 L | Loop invariant removal
157 O | Add offsets for use by the interpreter
158 P | Partitioning analysis
159 S | Stream analysis
160 V | Vector analysis
161 
162 ## Pragmas {#Pragmas}
163 
164 Pragmas are used to add additional information to an instruction. Currently, 2 types of pragmas exist, pragmas that are generated by the compiler, and pragmas that are added after analyzing. Multiple pragmas can be added after a single instruction. Pragmas are terminated by a whitespace character.
165 
166 Pragmas have the following syntax: `%<id>=<anything>,<otheranything> %<otherid>=<anything>`
167 
168 The following table provides an overview of the meaning of different pragmas. The type column indicates the type of pragma; C indicates a compiler-generated pragma, while A indicates a pragma generated by analyzing.
169 
170 Character | Type | Meaning
171 ----------|------|---------
172 bd | C | bounds
173 na | C | name
174 sl | C | source line
175 op | C | op number with line of source
176 ar | A | size of activation record needed
177 lz | A | edge carries a value that must be demanded
178 mk | A | mark this edge by reference (%mk=r) or by value (%mk=v)
179 of | A | offset in activation record
180 st | A | style of memory allocation (%st=p for pointer, %st=c for contiguous)
181 xy | A | position for node in graphic output

195 # Appendix C: Nodes {#Nodes}
196 
197 <!--
198 arith = integer real double
199 algeb = arith + bool

204 The next section desribes both the simple and compound nodes that IF1 predefines, a few datatype conventions should be established for this.
205 
206 * *arith* types correspond to any numeric type
207 * *algeb* types correspond to arith types and booleans
208 * *atom* types correspond to algeb types and characters
209 * *T* corresponds to any type
210 * `(T)+` Represents one or more occurence of T
211 * `(T)*` Represents zero or more occurence of T
212 * `[T]` Represents one or zero occurence of T
213 * `AS(T)` Represents an array or a stream that contains type T.
214 * `ASM(T)` Represents an array,stream or multiple that contains type T.

329 ### Select
330 
331 * Goal: The select node represents a multiway selection. Sisal only supports a 2 way selection, but this compound node could be used to implement different constructs such as a switch too.
332 * Subnodes:
333  * 1 predicate node that returns a number between 0 and n - 1, n being the amount of subnodes.
334  * Every other node is a path that can be followed depending on the result of the predicate.
335 * Association list:
336  * At least 3 associations (predicate, iftrue and iffalse in case of if)
337  * First element identifies predicate subgraph
338  * Every other element identify the subgraph to use for a given predicate result.
339 * Signature: `(value)+ -> (value)+`
340 
341 ### Forall
342 
343 * Goal: Independent execution of multiple instances of an expression
344 * Subnodes:
345  * **Generator:** Produce values for every instance of the body
346  * **Body:** Expression to be evaluated
347  * **Results:** Gathers the *ordered* results.
348 * Association list:
349  * Generator
350  * Body
351  * Returns
352 * Signature: `(value)+ -> (value)+`
353 
354 ### LoopA
355 
356 * Goal: Iterative looping construct. Stops when the *test* subgraph returns false. The test is executed after the body has executed once.
357 * Subnodes:
358  * Initialization
359  * Test
360  * Body
361  * Returns
362 * Association list:
363  * Same order as subnodes
364 * Signature: `(value)+ -> (value)+`
365 
366 ### LoopB
367 
368 * Goal: Iterative looping construct. Stops when the *test* subgraph returns false. The test is executed before the body is executed for the first time.
369 * Subnodes:
370  * Initialization
371  * Test
372  * Body
373  * Returns
374 * Association list:
375  * Same order as subnodes
376 * Signature: `(value)+ -> (value)+`
